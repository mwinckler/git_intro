The Adventurous Life of a Version Controller

or

by Matt Winckler
(mwinckler@intera.com)

3 Feb 2012

What is version control?

Version control is...

  • A way to keep track of changes in files
  • A structured way to collaborate between developers
  • Never having to say "I'm sorry"

Why should I use version control?

Some obvious reasons:

  • It keeps previous versions of files
  • It merges other people's changes for you
  • It shows a history of the code

Why should I use version control?

Some less obvious reasons:

  • It helps you fix bugs in deployed code
  • It gives you freedom to experiment
  • It can make audits easier to handle

What sorts of things should be controlled?

  • Everything that doesn't get created as part of your build process
  • ...and nothing that does

The traditional model of version control

  • Centralized version control systems rely on one central server keeping the "canonical" version.
  • There are several downsides to this.

What does "DVCS" mean?

  • A Distributed Version Control System is a VCS that doesn't rely on a central server.
  • Instead, each copy of the repository is a full repository--including history.
  • Distributed repositories can swap changes with each other.

No server?
What madness is this?!

No server. If this is madness, I don't want to be sane.

What about collaboration?

  • Well, what about it?
  • DVCS repositories communicate as peers.
  • You don't even need a special server for collaboration. (But it's good to have one.)

Why should I use a DVCS?

  • Speed: The Tortoise[SVN] versus the...Git
  • Redundancy: Each clone is a full repository
  • Mobility: Work on an airplane? No problem.
  • Scaling: Large numbers of files

Questions thus far?

Introducing Git

git-scm.com

What is it?

  • Git is distributed version control system software written by our favorite penguin-loving Finn, Linus Torvalds.

  • Think of it as Subversion without the server. And with better branching. And merging that won't make you old. And faster. And without .svn folders littered about the place.

  • But other than that, it's just like Subversion.

What's it good at?

Managing large numbers of text files...for instance, source code. (It was written for the Linux kernel.)

What's it bad at?

Handling enormously large files.

Possible remedy: git annex. Disclaimer: I don't know anything about the git annex project.

The Various And Divers States Of A File

A file with changes falls into one of four categories:

  • Untracked
  • Changed but unstaged for commit
  • Changed and staged for commit
  • Ignored

Untracked files

These are files that are present in your repository directory but that you have not told Git to keep track of (via git add).

By default, before ever running "git add", this is all your files.

Changed but unstaged

git status shows these as "Changes not staged for commit"

These are files that:

  • have been added to the repository,
  • have changed since the last commit, and
  • you have not yet told Git to include them in the next git commit

Changed and staged

git status shows these as "Changes to be committed"

These are files that:

  • have been added to the repository,
  • have changed since the last commit, and
  • you have told Git (via git add) to include next time you git commit

Ignored

  • git status will never show these.
  • These are files you've told Git to ignore via .gitignore (repo-based or global).
  • Git will never ignore files that have been added to the repository, even if they match a .gitignore entry.

Back on topic: committing

git add .
git commit

Switch to terminal, demonstrate

A commit shortcut

git commit -a

The a stands for "all" and will commit all changed files, staged or unstaged (but not untracked ones).

An even shorter cut

git commit -am "your message"

The m stands for "message" and lets you specify your commit message on the command line.

Rollbacks

What kind of rollback do you want?

  • Change the most recent commit?
    git commit --amend
  • Revert an entire commit?
    git revert <commit-hash>
  • Check out a particular file?
    git checkout <commit-hash> -- <file>
  • Don't know for sure? Find differences:
    git diff <commit1>..<commit2>

Amending the last commit

Go ahead and make the changes, then:

git commit --amend

DO NOT use --amend if you have already pushed to a remote repository! Your collaborators will curse your name!

Check out a particular version

git checkout <commit-hash> -- filename

retrieves filename as it existed in that commit

git checkout HEAD^4 -- filename

retrieves filename from 4 versions ago

Revert an entire commit

(the "Matt must have been drinking during that commit" option)

git revert <commit-hash>

Finding differences

git diff <file>

shows uncommitted changes in your working copy.

git diff <commit1>..<commit2> -- <file>

shows the differences in <file> between commit1 and commit2.

Collaboration Using Git

Collaboration Commands

  • git remote to view/change remotes
  • git push changes to a remote
  • git pull changes from a remote

What is a remote?

A remote is a repository located somewhere else that your Git client knows and cares about.

Creating Remotes

The easy way:

git clone <remote-location>

clone does two things: copies the remote repository and sets up a remote named origin so that you can push/pull changes to/from it.

Switch to terminal for demo.

Creating Remotes

The manual way:

git remote add <remote-name> <remote-url>

Do this if you already have a copy of the repository locally and you want to tie it to a copy at a remote location (or push it to a remote server).

Viewing remotes

To see what remotes you have set up, run:

git remote show

To get more details on a particular remote, use:

git remote show <remote-name>

Push and Pull

  • git pull retrieves changes from a remote and merges them into your working copy.
  • git push sends your changes up to a remote.

Push

Do this when you want to send your local changes up to a server for others to pull.

  • In order to push,
    1. Your working copy must be clean (no uncommitted changes)
    2. Your working copy must be up-to-date with the remote you're pushing to (no changes you haven't yet pulled)

Switch to terminal for demo.

Pull

Do this when you want to retrieve other people's changes.

  • In order to pull your working copy must be clean (no uncommitted changes).
  • pull performs the following actions:
    1. Retrieves changes from the specified remote
    2. Applies them to the local copy:
      1. If there are no local changes, "fast-forward" the remote changes
      2. If there are local changes, perform a merge

Pull

git pull <remote-name> <refspec>

Example: If I have a remote called origin and I want to retrieve the branch called master:

git pull origin master

Advanced Workflow

Git Advanced Workflow

  • git branch to work on stuff in isolation
  • git merge to get outside changes into your copy
  • git stash to tuck changes away temporarily
  • git tag to put a label on a version for easy lookup later
  • git rebase to destroy your sanity*


  • * results may vary

Branches

(image of branching goes here)

A branch is a copy of the code you can work on while leaving other branches untouched.

Branches

Each branch keeps track of the changes that have been made in it so that it can be merged back into the branch it came from, or vice versa.

Branches

Branches are cheap - Git only stores "deltas", not full copies. Make branches!

Creating a Branch

git checkout -b <branch-name>

Switching between branches

git checkout <branch-name>

Note: Your working directory need not be clean. This may or may not be desirable.

Switch to terminal for demo.

Merging

A merge grabs changes from a different branch and merges them into the current branch.

Merging

  • If the relevant files only changed on the branch being merged, it's a fast-foward merge.

  • If both files changed, it's a non-fast-forward merge, but Git might still do it without user interaction.

  • If both files changed the same lines, it's a conflict and requires user intervention.

Merging

First checkout the branch you want to merge onto, then:

git merge <branch-name>

Stashing

Forget about the changes in your working directory...temporarily.

Stashing

With unstaged changes in your working directory:

git stash

Warning: stash only stashes changes to files that Git already cares about.

Get the stash back

git stash pop

Note: This pops onto whatever your working directory currently is...even if its not the one you stashed from.

Branch from the stash

git stash branch <branch-name>

Tagging

A tag is a convenient way to label a commit.

You don't need a tag to retrieve old versions - but they can be handy if you know you'll need to reference a commit later.
